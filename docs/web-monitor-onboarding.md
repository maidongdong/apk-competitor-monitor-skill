# Web Monitor Onboarding

Use this guide when adapting the weekly monitoring pipeline to a new competitor Web/admin product.

## Goal

Create a stable `artifacts/web-monitor/config.json` for the target workspace so these MCP tools can run:

- `web_probe_admin`
- `web_generate_report`
- `web_run_weekly`
- `weekly_generate_unified_report`
- `run_full_weekly_pipeline`

## Files To Prepare

- Route config:
  - `artifacts/web-monitor/config.json`
- Auth storage state:
  - usually `artifacts/web-monitor/auth/admin.storageState.json`
- Baseline output:
  - usually `artifacts/web-monitor/baseline/admin-baseline.json`

Start from:

- [`apk-competitor-monitor/examples/web-monitor.config.example.json`](../apk-competitor-monitor/examples/web-monitor.config.example.json)

## What To Configure

Each `target` should define:

- `id`
- `name`
- `baseUrl`
- `authProfile`
- `storageState`
- `routes`

Each route should define:

- `id`
- `name`
- `url`
- `priority`
- `capture`
- `baselineSpec`

## How To Pick Routes

Prefer routes that reveal product capability changes instead of volatile operational data.

Good candidates:

- dashboard / stats pages
- list pages with stable filters and actions
- AI tool pages
- template / configuration pages
- admin entry pages with stable tabs or sections

Lower-value candidates:

- pages dominated by counts, dates, or transient rows
- pages that are almost entirely user-generated text
- pages that require too many account states to stabilize

## How To Write `baselineSpec`

Use `baselineSpec` to describe stable signals that should survive day-to-day operations:

- `stableControls`
- `tabs`
- `sections`
- `filters`
- `tableColumns`
- `rowActions`
- `options`
- `metrics`
- `categories`
- `formFields`
- `templateCards`
- `uploadConstraints`
- `highValueSignals`

Avoid stuffing unstable text into the baseline. Prefer:

- button labels
- section titles
- fixed tabs
- filter names
- column headers
- high-value feature labels

## Redaction And Noise Control

Use:

- `privacyRedaction`
- `ignoreTextPatterns`
- `rules.maskDynamicTextHints`

These reduce false positives and help avoid leaking customer/member/photo/location information into PM-facing outputs.

Recommended redaction for sensitive admin pages:

- `redactNames`
- `redactImages`
- `redactLocations`
- `redactMapTiles`

Recommended ignore patterns:

- dates
- times
- large counts
- autogenerated ids
- “更新时间” style labels

## First Run Workflow

1. Prepare `config.json`.
2. Capture login state in the target workspace.
3. Run `web_probe_admin`.
4. Inspect whether routes land on:
   - login page
   - team selection
   - blank page
   - network failure
5. Refine `baselineSpec` and ignore patterns.
6. Run `web_generate_report`.
7. Only then accept the first baseline.

## Practical Tips

- Start with 5-10 high-value routes before trying to cover the entire backend.
- Separate truly stable product signals from noisy operational text.
- Use route ids that remain readable in reports and artifacts.
- Treat login-state expiration as an operational failure, not a product change.
- Keep raw snapshots local; PM summaries should stay redacted.
